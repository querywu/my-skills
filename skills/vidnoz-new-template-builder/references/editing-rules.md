# 编辑规则

## 源码与运行时产物

优先视为源码层的目录：

- `templates/new-template/Dev/js/`
- `templates/new-template/Dev/scss/`

但模板实际加载的是：

- `templates/new-template/js/`
- `templates/new-template/css/`

构建流程在仓库外部。不要假设本地能重新编译运行时产物，也不要默认手工复制相同改动到运行时产物。

## 决策规则

- 如果存在匹配的 `Dev` 源文件：
  - 先改源码
  - 再检查模板实际引入的运行时产物
  - 默认不手工同步同名运行时产物
  - 如果运行时产物看起来没有更新，提醒用户确认外部编译器是否已经开启
- 如果不存在匹配的 `Dev` 源文件：
  - 直接修改运行时产物
  - 明确说明当前改动以运行时文件为准
- 如果一个运行时文件明显由多个共享模块拼出来：
  - 修改最小相关的共享源码或共享运行时文件
  - 避免对无关模块做猜测式重写

## 安全的优化类型

可以做的优化：

- 删除当前页面里重复的选择器
- 收敛当前页面里重复的 DOM 查询或事件绑定
- 复用当前页面已存在的共享工具方法
- 只在确认是本需求范围时，删除无用语言 key 或冗余模板结构
- 在不扩大范围的前提下，顺手修正明显影响 SEO 的实现问题，例如关键文案仅靠 JS 注入、标题层级错乱、图片缺少必要 `alt`

除非明确要求，否则避免：

- 全局命名重写
- 从 jQuery 批量迁移到其他框架
- 跨页面设计系统重构
- 修改发布控制器行为
- 改造与当前需求无关页面的 `lan.json` 结构

## SEO 实现边界

这个项目默认重视 SEO，但本 skill 的 SEO 关注点是页面实现层，不是全量 SEO 基建。

默认应关注：

- `tpl` 中是否直接输出核心主题文案与主要内容块
- 标题层级是否清晰
- 关键图片是否具备合适的 `alt`
- `js` 是否破坏正文可见性、首屏文案或核心区块结构

默认不纳入本 skill 工作范围：

- `canonical`
- `alternate`
- 结构化数据

## 发布安全

不要破坏以下前提：

- 页面模板文件名与页面语言 key 保持对应
- 预览与发布控制器仍能正常读取模板
- 模板中的相对资源路径仍有效
- 共享 `header` 与 `footer` 仍能用共享语言对象正常渲染

## 外部编译器提醒

当任务修改了 `Dev/js` 或 `Dev/scss` 后，发现页面实际引用的 `js/` 或 `css/` 产物没有同步变化时：

- 不要直接复制一份相同改动到产物层
- 先提醒用户确认外部编译器是否已开启
- 只有在根本不存在对应源码文件时，才直接编辑运行时产物
